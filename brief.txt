## AUTHENTICATION

### Passkey (WebAuthn)
- Use `@simplewebauthn/server` on the API routes and `@simplewebauthn/browser` on the client.
- Registration flow: `POST /api/auth/passkey/register` → returns options → browser creates credential → `POST /api/auth/passkey/verify`.
- Authentication flow: `POST /api/auth/passkey/authenticate` → returns options → browser signs challenge → `POST /api/auth/passkey/verify`.
- Store credential public key, counter, and transports in the User model.

### Email OTP
- On login form submit (email entered), call `POST /api/auth/otp/send`.
- Generate a 6-digit OTP with `otplib`, hash it, store with 10-minute expiry on the User document.
- Send via SendGrid using the template in `src/lib/sendgrid.ts`.
- Verification screen (`/verify`) accepts the 6-digit code → `POST /api/auth/otp/verify` → issue NextAuth session.

### Login Page UX
- Two clearly labelled tabs: **"Sign in with Passkey"** and **"Email Code"**.
- Passkey tab: single "Continue with Passkey" button, animated fingerprint/key icon via Framer Motion.
- OTP tab: email input → submit → transitions (Framer Motion `AnimatePresence`) to 6-box OTP input.
- Show inline feedback for every state: sending, sent, wrong code, expired, success.
- Never redirect without a toast/notification confirming what just happened.

## ADMIN ONBOARDING FLOW

This is an admin-only wizard for setting up a new client. It must cover:

1. **Invite** — Enter client name + email → send branded onboarding invitation email via SendGrid → client record created with status `invited`.
2. **Intake on behalf** — Admin can fill the intake form for the client (mark `completedBy: 'admin'`). The intake form sections and field order must exactly match what was extracted from the HTML in Step 1.
3. **Assign package** — Dropdown of coaching/consulting packages → creates the first Project record linked to this client.
4. **Review & confirm** — Summary screen showing all entered data before saving.

Show a persistent **onboarding progress indicator** (step dots or a sidebar stepper with Framer Motion transitions between steps).

## PROJECTS OVERVIEW

The projects page is the operational heart of the portal:

- **Kanban-style or table view** (toggle between them) showing all active projects.
- Each project card shows: client name, package/type, current status badge, days active, assigned consultant.
- **Block alerts** — if `status === 'blocked'`, render a prominent amber/red alert on the card showing the block reason and how long it has been open. The Projects page header must show a running count of currently blocked projects.
- **Filters**: by status, by consultant, by date range.
- Clicking a project opens a slide-over panel (Framer Motion `motion.div` from the right) with full detail, timeline of status changes, and block history.
- Admin can change status, add/resolve blocks, and add notes directly from the panel without navigating away.

## UX & NOTIFICATION REQUIREMENTS

This is a top priority. Every user action must have clear, immediate feedback.

### Toast Notifications
- Implement a global toast system (top-right, stacked).
- Use Framer Motion for enter/exit animations.
- Four variants: `success` (green), `error` (red), `warning` (amber), `info` (blue).
- Auto-dismiss after 5 seconds with a visible countdown progress bar on the toast.
- Always fire a toast after: login, logout, form save, form submit, status change, email sent, OTP sent, block raised, block resolved.

### Inline Form Feedback
- Real-time Zod validation on blur.
- Field-level error messages animate in with Framer Motion.
- On multi-step intake: show a section completion indicator (checkmark animates in when a section is fully valid).

### Empty States
- Every list and table has a designed empty state (icon + message + CTA where appropriate).
- Use Framer Motion fade-in on empty state render.

### Loading States
- Skeleton loaders for all data-fetching components.
- Button loading states (spinner replaces icon, button disabled) during async operations.

### Confirmation Dialogs
- Destructive actions (delete client, resolve block, etc.) require a modal confirmation.
- Modal uses Framer Motion scale + fade entrance.

## DESIGN SYSTEM & AESTHETIC

Create a **refined, professional** design language appropriate for a high-end business consulting firm:

- **Palette**: Deep navy (`#0F1F3D`) primary, warm white (`#FAFAF8`) background, slate greys for secondary text, a single sharp accent — gold (`#C9A84C`). Define all as Tailwind CSS variables in `tailwind.config.ts`.
- **Typography**: Use `DM Serif Display` for headings paired with `DM Sans` for body text. Load both via `next/font/google`.
- **Spacing**: Generous whitespace. Cards with subtle box-shadow, not heavy borders.
- **Sidebar**: Fixed left sidebar, collapsible to icon-only. Shows portal logo, nav links with animated active indicator, user avatar + name at bottom, logout button.
- **Motion principles**: Page entrance — subtle fade + slight upward translate (300ms ease-out). Staggered list items. Hover states on all interactive elements. Keep animations purposeful, not decorative.
- **Avoid**: Purple gradients, Inter/Roboto, rounded pill buttons, generic SaaS blue, cluttered layouts.

## CRITICAL RULES — NEVER VIOLATE THESE

1. **Never alter any text from the HTML** — not a word, not punctuation, not capitalisation.
2. **Never reorder information-gathering sections** — the sequence is intentional and client-approved.
3. **Every user action must produce visible feedback** — no silent operations anywhere.
4. **Auth routes must be protected at middleware level** — not just client-side redirects.
5. **MongoDB connection must use connection caching** — never open a new connection per request in serverless.
6. **All forms must have Zod schemas** — no unvalidated data reaches the database.
7. **Passkey and OTP are both first-class auth methods** — do not deprioritise one over the other.
8. **The `_temp/` folder and the HTML file inside it must never be deleted.**
